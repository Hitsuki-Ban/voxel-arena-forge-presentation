「原型箱」を先に作る

LLM にコードを書かせようとするとき、最初に考えがちなのは「どのモデルを使うか」です。 VoxelArenaForge では順番を逆にしました。先に「型」を作り込み、モデルはあとから安いものに差し替えていく。 この型の一式を、私たちは「原型箱」と呼んでいます。

第1回で紹介した「文章からゲームユニットを作る」体験の裏側で、生成 AI に何を渡しているのかを書きます。 シリーズの重点回その1です。

原型箱の中身

VoxelArenaForge の生成対象は、ユニットの行動スクリプト(TypeScript)とボクセルモデルの構成です。 LLM に渡している「箱」は、大きく4種類あります。

1. コマンド API — できることのメニュー

生成スクリプトは、世界の状態を直接いじれません。できるのは 16種類の Command を返すことだけです。少ないので全部書きます:

  • 装備・移動: equip / mount / attachAura / moveToward / faceTarget
  • 攻撃: meleeArc / areaDamage
  • 妨害・支援: areaPull / areaSlow / applyStatus / areaStatus / applySelfStatus / imbueSelf
  • 演出・管理: playAttributeVfx / playVfx / startCooldown

引数も型で固定です。たとえば areaDamage は { center, radius, damage, damageType, maxTargets?, falloff?: 'none' | 'linear', vfx? } — falloff に数値を書くと型検査で弾かれます。

見える情報(ActionContext)も同じ思想で絞ってあります。 自分の位置と HP 比率(self)、AI テンプレートが決めた攻撃対象・目標地点(intent)、 attack と moveSpeed の2つの能力値(scalars)、 cooldowns.ready(id)、シード付き乱数(rng.next())— これで全部です。 敵の一覧も、HP の生値も、座標系の詳細も渡しません。

つまり LLM への依頼は「自由にコードを書いて」ではなく、「このメニューから技を組み合わせて」になります。 書ける範囲と見える範囲が狭いほど、出力は型に沿い、検証も簡単になります。

2. スキーマ — データの通り道

ユニット仕様、生成パッケージ、対戦ログなど、システムを流れるデータはすべてスキーマ(23ファイル)で定義され、 検証を通らないデータはそもそも次の工程に進めません。属性は8種、状態異常は11種と列挙型で固定。 LLM が「それっぽい何か」を発明する余地を削っています。

3. 素材カタログ — 存在するものしか使えない

ボクセルプリミティブ12種、3Dアセット43種(体・武器・騎乗・VFX)。 生成コードが参照できるのはカタログに実在する ID だけで、存在しないエフェクト名を書くと受け入れ検査で弾かれます。 行動の骨格も、7種の AI テンプレート(Brawler / Kiter / Artillery / Diver / Support / Summoner / Boss)から選ぶ方式です。

4. 参照ドキュメント — 動く実例と、失敗の記録

生成プロンプトには6本の参照文書が同梱されます。その中で特に効いたのが次の2本でした。

  • working-heavy-swordsman-example — 検証を通った実物のスクリプトを丸ごと渡す。「良い出力の説明」より「良い出力そのもの」の方がずっと効きます。
  • known-failure-patterns — 過去に検証で弾かれた失敗パターン集。失敗が発生するたびにここへ蓄積され、次の生成プロンプトに反映されます。失敗が資産になる設計です。
腐らないドキュメント。 この参照パックはリビジョンとハッシュで管理され、記載内容が現在のリポジトリの実体(コマンド定義・カタログ)と 一致するかをテストが毎回機械検証します。コードを変えたのにドキュメントを直し忘れると、CI が落ちます。 人間のドキュメントが腐る問題を、契約として解決しています。

で、モデルはどこまで安くできたか

原型箱を作り込んだ結果を、実測の数字で書きます(受け入れ率=生成物が全自動検査を通ってライブラリ登録まで到達した割合です)。

  • 仕様相談(仕様相談 AI)は軽量級の deepseek-v4-flash で成立しています。
  • コード生成を中量級(pro)にした構成は、調整済みの表示用10件で 10/10(初回成功率 1.0)、調整に使っていない未見の holdout 10件でも 9〜10/10。
  • そして本題の、コード生成まで flash にした軽量単独構成。原型箱が未成熟だった初期の計測では 25% でした。それが型の成熟した現在は、表示用10件で 10/10(初回成功率 0.9)、未見の holdout 10件でも 10/10(初回成功率 1.0)です。
  • 履歴全体では 1,360 回の生成試行で受け入れ率 75.7%。これは調整前の失敗も全部含んだ数字で、175本の評価レポートから改善の推移がそのまま追えます。
  • コストの実測も付けておきます。受け入れ1体あたり flash 単独で約14,100トークン(入力11,600/出力2,600)、pro 構成で約13,600トークン。トークン量は構成でほぼ変わらないので、コスト差は純粋にモデル単価の差です。
25% → 10/10同じ軽量モデルの受け入れ率。モデルは一度も変わっていない
10/10未見の holdout でも通過(初回成功率 1.0)
約1.4万受け入れ1体あたりのトークン。構成間の差は4%未満

同じ軽量モデルが 25% から 10/10 になるまでの間に、モデルは一度も変わっていません。変わったのは型と失敗パターン集です。 そしてこの実験を安心して回せた理由も型の側にあります。flash の出力が暴れても、受け入れ検査が全部止めてくれる。 壊れたものは一切通らないので、「モデルをどこまで安くできるか」が完全にノーリスクな実験になる — これがこの記事で一番言いたいことです。

正直に書くと、順序を間違えた時期もありました。初期に重量級(pro/pro)構成で回した本番スモークは惨憺たる結果で、 モデルを重くしても解決しませんでした。数字が動き始めたのは、known-failure-patterns と working example を整備してからです。 モデルの賢さは、型の不備を埋めてくれませんでした。

スクリプトのかたち — 「頭脳」と「技の中身」の2層

実物のコードを見せる前に、その「かたち」を説明させてください。プログラムに馴染みがなくても、ここさえ分かれば読めます。

まず大事な前提として、ユニットの動きは2つの層に分かれています。

  1. ユニット AI(頭脳)— 生成しません。 「いつ動くか」「誰を狙うか」「今どの行動を出すか」を毎瞬決める層です。 これは素材カタログの項で触れた7種の既製テンプレート(接近して殴る Brawler、距離を取って撃つ Kiter、 後方から砲撃する Artillery…)から選ぶだけで、中身はエンジン側の検証済みの部品です。
  2. 行動の指示書(スクリプト)— こちらを生成します。 頭脳が「今スキル1を使う」と決めた瞬間に呼び出され、その技が具体的に何をするかを答える層です。 LLM が書くのはこの指示書だけです。

この分業は原型箱の思想そのものです。戦い方の頭脳を既製品にしておけば、 生成がどれだけ暴れても「棒立ちになる」「戦場を迷走する」といった壊れ方は構造的に起きません。 生成の自由度を「技の中身」に閉じ込めているわけです。

その指示書は、場面ごとに10枚あります。 出撃したとき(init)、移動の表現(move)、向きの表現(turn)、 通常攻撃・強攻撃のとき(basicAttack / heavyAttack)、スキルを使うとき(skill1 / skill2)、 攻撃を受けたとき(onHit)、転倒したとき(onKnockdown)、倒れたとき(onDeath)。 舞台の台本にたとえるなら、頭脳が「次はこの場面」とページを指定し、指示書がそのページの中身を書いている関係です。

どのページも、書いてあることは同じ構造です。

  1. 状況メモ(ctx)を受け取る。自分の位置とHP割合、頭脳が決めた攻撃対象、「スキルはもう使えるか」など、いま見えている情報のメモです。
  2. 命令カードを数枚選んで返す。先に書いた16種のメニュー(範囲減速、クールダウン開始…)から選ぶだけ。
  3. 実行はシミュレーターがやる。指示書自身は世界に指一本触れられません。返した命令が正しいかどうかも、実行前に検査されます。

つまりこれから見るコードは「プログラム」というより、様式の決まった記入用紙に近いものです。 skill1: … は「スキルを使うときの指示書」、return [...] は「返す命令カードの束」— この2つだけ押さえてください。

実際に出てきたものを見る

数字だけだと実感が湧かないと思うので、実物を貼ります。調整に使っていない holdout スイートの依頼文 「鎖付きの槍を操る騎兵ユニットを作ってください。騎乗感、長柄武器、鎖で距離を支配する動きを見せてください。」に対して、 軽量モデル(flash)が初回一発で出したコードです(無修正の抜粋)。

script.tsflash が初回一発で生成(無修正・抜粋)
export const unitScript: UnitScriptV1 = {
  version: 1,
  init: () => [
    commands.equip('chain_snare_tool'),
    commands.equip('rider_lance_polearm'),
    commands.mount('void_horse')
  ],
  // …move / turn / basicAttack 略…
  skill1: (ctx: ActionContext) => {
    if (!ctx.cooldowns.ready('skill1')) return [];
    const output: Command[] = [
      ctx.commands.areaSlow({
        center: ctx.intent.targetPosition ?? ctx.self.position,
        radius: 4,
        slowRatio: 0.5,
        duration: 2,
        vfx: 'chain_tether_spark'
      }),
      ctx.commands.startCooldown('skill1', 7)
    ];
    return output;
  },
  // …skill2 / onHit / onKnockdown / onDeath 略…
};

前章の読み方で追うと: init(出撃したときの指示書)では鎖・槍を装備して馬に乗り、 skill1(スキルの指示書)では「まだ使えないなら何もしない(ready の確認)」→ 「範囲減速の命令と、クールダウン開始の命令をカード2枚として返す」— それだけです。

見てほしいのはコードの賢さではなく、語彙の狭さです。 equip / mount の ID は全部アセット台帳の実在 ID(chain_snare_tool、void_horse)。 areaSlow の引数は型どおり。クールダウンは ready() を確認してから startCooldown で開始。 flash 級のモデルにこれが書けるのは、書ける範囲をここまで絞ってあるからです。

1回の生成で出てくるのは、このスクリプトを含めて5ファイルです。

  • script.ts — 行動スクリプト(上の実物)
  • manifest.json — アクション定義。射程・クールダウン・対象選択・効果範囲の予算(このユニットの skill1 なら range 4 / cooldown 7 / targeting "position" / budgets { maxRadius: 4, maxTargets: 3 })
  • composition.json — ボクセルの見た目構成(体・槍・騎乗)
  • tests.generated.ts — 生成物自身の形状テスト
  • brief.ja.md — 日本語の設計解説

対戦後には balance-report が付きます。このユニットなら平均 DPS 198.6、平均生存 17.0 秒 — 「なんか強そう」ではなく数値で残るので、バランス調整の入力にそのまま使えます。

生成物は完成品でなくても「叩き台」になる

ここまで「全自動で受け入れる」話をしてきましたが、この様式にはもうひとつ実務的な効果があります。 人がそのまま読めて、そのまま直せることです。

指示書は決められた命令カードの組み合わせなので、直す場所がいつも同じです。 範囲を狭めたければ radius: 4 を 3 に、回転を遅くしたければ startCooldown('skill1', 7) を 8 に、 エフェクトを替えたければ vfx の ID を台帳の別の ID に。 プログラムの内部構造に詳しくなくても、数字と ID の差し替えでバランス調整ができる形になっています。

だから生成が一発で理想形に届かなくても、工程は前に進みます。白紙から書くのではなく、叩き台を直す仕事になるからです。 そして人が手を入れた結果も、AI の出力とまったく同じ受け入れ検査と対戦シミュレーションを通ります。 「誰が書いたか」ではなく「検査を通ったか」で品質を担保しているので、 人と AI が同じ様式の上で交互に手を入れても、保証は崩れません。 前章の balance-report の数値は、この「直す」ループの入力としてそのまま機能します。

ゲーム開発現場への翻訳

「原型箱」と呼んでいるものは、ゲーム開発の現場ならすでに持っているもので作れるはずです。

この実験での型現場での対応物
コマンド API(16種)スキルやアクションのプロジェクト内 API・コマンド定義
スキーマ(23本)マスタデータのスキーマ、エディタの入力規則
素材カタログ(43点)アセット管理台帳、エフェクト・モーションの ID 一覧
参照ドキュメント(6本)通った実データのサンプルと、レビューで弾いた理由の記録

新しく作るものより、「すでにあるものを LLM が読める契約に固め直す」作業が中心です。 そして型が固まった瞬間から、高いモデルを安いモデルに差し替える実験が、数字付きで安全にできるようになります。
