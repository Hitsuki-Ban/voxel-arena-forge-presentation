LLMに「作らせて、検証する」 第5回 / シリーズ目次

AIエージェントと6週間、650コミット

ここまでの4回は「LLM にゲームコンテンツを作らせる」話でした。この回はメタな話です — その実験機自体を、ほぼ AI コーディングエージェントが実装しました。 2026年5月24日の初コミットから、最初の3週間で618コミット、発表準備も含めた現在で約650コミット。 人間が書いたコードはごくわずかです。

うまくいった体制と、実際にあった事故を両方書きます。事故の方が参考になるかもしれません。

体制 — 誰が何をしたか

登場人物は3者です。

  • 人間(オーナー) — 何を作るかの決定、フィードバック、予算とタイミングの判断。コードはほぼ書かない。
  • 実装エージェント(Codex) — ロードマップの粒度の実装。タスクを受けて、実装・テスト・進捗ログをまとめた変更提案(ドラフト PR)を出す。
  • レビュー/資料エージェント(Claude) — 変更提案のレビューと取り込み、作業依頼(Issue)の起票、発表資料、フロントエンド系の骨組み実装。

運用の型は「Issue → ドラフト PR → レビュー → マージ」の普通の開発フローで、 特別なのは各役をエージェントが担っている点だけです。途中からはフロントエンドが重いタスクについて 「片方が骨組みを作り、もう片方が完成させる」分業も使うようになりました。 スコアボードや決着バナーの類いは骨組み→完成の2段で入っています。

ロードマップと進捗ログ

リポジトリには CODEX_TASKS.md というロードマップがあり、T00(開発環境の初期化)から T18(演出層 QA)まで、各タスクに「Goal / Deliverables / Acceptance」と日付つき進捗ログが積まれています。 エージェントは毎回このログに「何を変えて、どう検証したか」を書き足してから終わります。 人間のスタンドアップの代わりに、ログがそのまま開発史になっています(現在 T438 番台まで)。

なぜ回ったのか — 検査は人間のためでもあった

第4回で書いた受け入れ検査と QA ゲートは、生成ユニットのためのものでした。 ところが実際に一番恩恵を受けたのは、エージェント開発そのものです。

  • 単体テスト約2,000本+画面を通しで動かす自動テスト(E2E)80本超が常時回る。エージェントの変更が何かを壊せば、取り込み前に機械が止める。
  • 決定論シミュレーションのおかげで「変更前後で対戦結果が変わっていないか」をハッシュ比較で検証できる。
  • ドキュメント契約テスト(参照パックとリポジトリの一致、コマンド説明の存在確認)が、エージェントが書き散らかしがちなドキュメントの腐敗を止める。

エージェントは疲れないかわりに、確認を省略しがちで、自信満々に間違えます。 その前提に立つと、「生成物を信用しない設計」と「開発者を信用しない設計」は同じものでした。 人間のチームのために作られてきた自動ビルド・テスト・レビューの文化は、エージェント開発にそのまま効きます。 むしろ、遠慮なく厳しくできるぶん効きが良いくらいです。

実際にあった事故簿

順調な話だけでは参考にならないので、起きたことをそのまま書きます。

事故何が起きたか対策
コミットの迷子 同じチェックアウトで2つのエージェントが作業し、片方のブランチ切替に巻き込まれて、もう片方のコミットが main に乗ってしまった エージェントごとに git worktree を分離。コミット直前のブランチ確認を手順化
改行コードで110テスト失敗 Windows 環境の CRLF 変換が参照パックのハッシュ検証を壊し、一見無関係な大量のテストが落ちた リポジトリを LF 固定(autocrlf=false)。原因は3分、特定は30分の典型例
ドキュメントの時限爆弾 テスト用の固定データに日付が直書きされており、30日の保持期限を過ぎた日から2本のテストが落ち始めた 相対日付に修正。「時間が経つと落ちるテスト」はエージェントも書く
録画中の開発サーバー死 長時間のフレーム撮影中に Vite が突然落ち、撮影バッチが途中で止まった 撮影ツールにヘルスチェックと自動再起動・再開を実装(これもエージェントが実装)
Windows のファイルロック 撮影後の一時ファイル掃除が EBUSY で失敗し、成果物の記録(マニフェスト)ごと失われた 記録の書き込みを掃除より先に移動+リトライ。順序で解決する種類の問題だった

共通しているのは、どれも「エージェントが悪い」種類の事故ではないことです。 並行作業・環境差・時間経過 — 人間のチームでも起きる問題が、速度が上がったぶん早く顕在化しただけでした。 そして対策もすべて、仕組み(worktree、設定固定、リトライ、ヘルスチェック)で入れています。
