LLMに「作らせて、検証する」 第4回 / シリーズ目次

LLMの出力を信用しない、という設計

このプロジェクトの品質保証には、人間のコードレビューが存在しません。 代わりに、生成物が通らなければならない検査の層を重ね、その通過率を数字として記録し続けています。 この記事はその「弾く側」の中身と、弾く仕組みがあったからこそ取れた計測の記録です。

発表スライドの数字(25%→10/10、1体あたり約1.4万トークン)の出どころを、ここで全部示します。

受け入れ検査の4層

  1. サンドボックス — そもそも触れない。 生成スクリプトは世界の状態を直接変更できず、16種のコマンド列を返すことしかできません(第2回)。 見える情報も自分の位置・HP比率・攻撃対象・能力値2つ・クールダウン確認・シード付き乱数だけ。 「悪さのしようがない」形を先に作ってあります。
  2. 静的検査 — 実行する前に弾く。 禁止 import・グローバル参照、アセット台帳に存在しない ID、マニフェストの記述漏れ、 型検査(TypeScript コンパイラをそのまま検査器に使います)。 第3回で見たとおり、ここで弾かれたエラーは人ではなく AI に返送されます。
  3. 生成テスト — 生成物が自分を検査する。 生成物には形状テスト(tests.generated.ts)が同梱され、構造の自己検証をします。 テスト自体もルール(ループ禁止・ハンドラ呼び出し禁止)で縛り、テストによるごまかしも塞ぎます。
  4. 決定論シミュレーション — 実戦で確かめる。 シード付き乱数と固定の処理順で、同じ入力なら対戦ログがバイト単位で同一になります。 生成ユニットは既存ユニットと実際に戦い、勝率・与ダメ・生存時間・行動阻害時間が数値で残ります。 描画(Three.js)は結果に影響しない「表示だけ」の層です。
ゲームを題材に選んだ理由はここです。 決定論シミュレーションという「安くて速くて機械的な最終検証」が作れる領域だから。 LLM の出力の良し悪しを、感想ではなく再現可能な実験として議論できます。

検査の上に載る QA ゲート

個々のユニットを弾く検査の上に、開発・展示の節目で使う QA ゲートが載っています。 代表的なものだけ挙げると:

  • 10件生成ゲート — 決まった10件のお題を実 API で流し、全件が受け入れ検査を通るかを見る合否判定。調整に使う「表示用スイート」と、調整には一切使わない「抜き打ちスイート」(holdout)を分けています。
  • 生成スクリプト整潔度 — 通ったコードの読みやすさ(長すぎる行、空のハンドラ、クールダウン漏れ、装備とタグの不一致)を別枠で採点。
  • デモ準備度 — サンプル台帳 → LLM 接続先の疎通 → 10件生成 → プレビュー描画確認を一括で回し、「今この場で見せられるか」を1つの証拠ファイルにまとめます。

どのゲートも証拠ファイル(JSON+Markdown)を残し、スキーマ・作成時刻・コミット一致まで検証されます。 古い証拠や壊れた証拠を「最新の成果」として読んでしまう事故を、仕組みで防いでいます。

ドキュメントも検査対象です。 生成プロンプトに入る参照パックは「記載内容がリポジトリの実体と一致するか」をテストが検証しますし、 QA コマンドの説明文すら「ドキュメントに書いてあるか」をテストが読みに行きます。 実際、この記事群の執筆中に README を再構成したら、ドキュメント契約テストがすぐに教えてくれました。

計測の記録 — 25% から 10/10 へ

検査が全部止めてくれるので、モデルや型を変える実験はノーリスクです。 壊れたものは絶対にゲームへ入らない。だから安心して数字を取り続けられます。その記録がこれです。

時点構成結果何を変えたか
初期軽量単独(flash/flash)25%(5/20)—(型が未成熟)
初期重量級(pro/pro)惨憺モデルを重くしても数字は動かず
中期flash + pro改善開始失敗パターン集と動く実例を整備
現在flash + pro10/10(初回 1.0)型の成熟のみ。モデルは不変
現在軽量単独(flash/flash)10/10(初回 0.9、抜き打ち 10/10)

言い換えると、合格率を決めていたのはモデルの賢さではなく、型と失敗の蓄積でした。 履歴全体では 1,360 回の生成試行・受け入れ率 75.7%(調整過程の失敗を全部含む)で、 175本の評価レポートから推移がそのまま追えます。

コストの計測 — 1体いくらか

全ランにトークン使用量を記録する配線を入れ、同じ10件のお題を両構成で流した実測です(2026-07-07、各10/10受理)。

14,128flash 単独: 受け入れ1体あたりトークン(入力11,558/出力2,570)
13,623pro 構成: 受け入れ1体あたりトークン(入力11,253/出力2,370)
<4%構成間のトークン量差。コスト差はほぼ純粋にモデル単価差

注目してほしいのは、トークン消費量が構成間でほぼ同じだという点です。 つまり「安いモデルに替えたら余計に呼び出し回数が増えて結局高くつく」ことは起きていません。 検査が品質を 10/10 に担保している限り、単価の安いモデルへ替えた分がまるごと削減になります。 内訳を見るとコスト の大半は実装側(コード生成)で、相談側は全体の15%程度でした。

まとめ

  • 品質保証は人手レビューではなく、4層の受け入れ検査(サンドボックス・静的検査・生成テスト・決定論シミュレーション)で担保する。
  • 検査があるから、モデルと型の実験がノーリスクになり、25%→10/10 のような推移を安全に実測できた。
  • コストは受け入れ1体あたり約1.4万トークン。トークン量は構成間でほぼ同一なので、軽量化の単価差がそのまま効く。
