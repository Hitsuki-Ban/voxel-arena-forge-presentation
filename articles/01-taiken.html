LLMに「作らせて、検証する」 第1回 / シリーズ目次

文章から、動くゲームユニットが生まれるまで

「鎖付きの槍を操る騎兵。距離を支配し、敵を足止めする。」— この一文をチャット欄に入れると、数分後には 3D のボクセルモデルを持ったユニットが画面の中で実際に戦い始めます。 行動ロジックのコードも、見た目も、バランス評価も、すべて自動生成です。

この記事は、実験プロジェクト VoxelArenaForge の体験を入口に、 「LLM に成果物を丸ごと作らせる」とはどういうことかを紹介するシリーズの第1回です。

何を作ったのか

VoxelArenaForge は、3つのアプリでできています。

  • ユニット工房(Workbench) — 日本語で「作りたいユニット」を相談する場所
  • シミュレーター(Simulator) — 生成されたユニットの対戦リプレイをブラウザで再生する場所
  • サーバー — 裏側で LLM への依頼と、生成物の検証を回す部分

題材はゲームのユニット作りですが、本当に実験したかったのは 「LLM の出力を、人が逐一チェックしなくても受け入れられる仕組み」の方です。 ゲームを選んだ理由は後で出てきます。

体験してみる

工房を開くと、チャットと「仕様シート」が並んでいます。冒頭の一文を送ると、AI(仕様相談 AI)がこう返してきます。

アシスタント: このユニットは主にどの役割にしたいですか?

そして「おすすめ」「攻めた案」「変化球」と書かれた3枚の候補カードが並びます。 範囲制御に寄せるか、前衛タンクにするか、近接アタッカーにするか。 それぞれに「見返り」と「リスク」が書いてあるので、選ぶだけで会話が進みます。

ここが最初の設計ポイントでした。自由入力だけの対話は、実は使う側の負担が大きい。 迷う場面では選択肢をカードにして出す方が、人の意思決定はずっと軽くなります。

仕様が固まると、シートに役割・コスト・ステータス・アビリティが埋まっていき、生成が始まります。 数分待つと、ユニットの行動スクリプト、ボクセルモデルの構成、そしてテストまでが揃って出てきます。

完成の定義は「戦えること」

生成が終わると、そのユニットはすぐに既存ユニットと対戦させられ、リプレイが再生されます。 勝敗、与ダメージ、生存時間。「なんとなく良さそう」ではなく、数字と映像で結果が返ってきます。

このプロジェクトでは、完成の定義を「コードが生成されたこと」ではなく 「検証を通ったこと」に置いています。生成されたスクリプトは安全性の検査を通り、テストを通り、 シミュレーションで実際に動いて、初めてライブラリに登録されます。

そしてこの対戦シミュレーションには大事な性質があります。 同じ条件なら、必ず同じ結果になる(決定論)ことです。 これが何を意味するかは第4回でじっくり書きますが、一言でいえば 「LLM の出力の良し悪しを、再現可能な実験として議論できる」ようになります。

正直に言うと

順風満帆に見えますが、実際には生成がうまくいかないことも普通にあります。 存在しないエフェクトを参照しようとしたり、クールダウン処理を書き忘れたり。

面白いのは、その失敗への対処法でした。失敗パターンを人が直すのではなく、 「やってはいけないことリスト(参照ドキュメント)」として蓄積し、 次の生成時にプロンプトへ自動で混ぜ込みます。失敗が資産になる設計です。 それでも通らないものは、受け入れ検査が容赦なく弾きます。

数字でみると

1,534回実 API でのユニット生成ラン(検証済み成果物として保存)
175本品質を継続計測した評価レポート
約3週間このプロジェクト自体の開発期間。ほぼ AI コーディングエージェントが実装(第5回)

このシリーズについて

  1. 文章から、動くゲームユニットが生まれるまで(本記事)
  2. 「原型箱」を先に作る — 軽量 LLM に整形済みスクリプトを出させる方法
  3. 作成ツールと LLM の「やり取り方式」 — 質問ゲート・候補カード・再試行フィードバック
  4. LLM の出力を信用しない、という設計 — 受け入れ検査と計測
  5. AI エージェントと3週間、700コミット — エージェント駆動開発の記録
  6. で、ゲーム開発のどこから使うか — この実験からの示唆

ゲーム開発の現場で「LLM に何をさせるか」を考えている方の、議論の材料になれば幸いです。
