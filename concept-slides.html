ゲーム開発 × LLM 運用の実証

軽量LLMに、
コンテンツを量産させる

日本語の一文から「検証済みの動くゲームユニット」を作る実験機 VoxelArenaForge でわかった、
安いモデルを現場で回すための作り方の話

先に結論

① 高価なモデルは必須ではない。「型」と「自動検査」を作り込めば、 軽量モデルだけの構成でも量産が回る(型の成熟前 25% → 現在 10/10)

② 品質保証は人手レビューではなく、仕組みで担保できる。 検査を通らない生成物は一切ゲームに入らない。だから安いモデルの実験がノーリスクになる

③ 原料は既存資産で足りる。 仕様書のテンプレート・マスタデータ定義・アセット台帳 — これを LLM が読める「型」に固め直す作業が本体だった

実証の題材: VoxelArenaForge

  • 「鎖付きの槍を操る騎兵ユニットを作ってください。騎乗感、長柄武器、鎖で距離を支配する動きを見せてください。」— 入力はこの日本語だけ(実測バッチで使った実際の依頼文)
  • AI が対話で仕様を固め、行動スクリプト・3D モデル構成・テストを自動生成
  • 生成物は自動検査と対戦シミュレーションを通ってはじめて登録される — 完成の定義は「検証を通ること」

題材はオートバトルのユニット制作。狙いは「LLM 生成物を安全に量産ラインに乗せる方法」の実証

まず、できあがるもの

画面で起きていること:

  • 一体一体がボクセル製ユニット(体・兜・武器・騎乗)。足元の板がチーム色(青=自軍/赤=敵軍)
  • 上部は生存数スコアボード、閃光は属性 VFX(雷)。決着時には勝敗バナーが出ます
  • 動き・攻撃・勝敗はすべて決定論シミュレーションの記録の再生 — 同じ条件なら同じ結果

生成ユニットの多様性(ベスト盤)

実 API で生成された 1,534 ランからの選抜。すべて検証済みで、その場でリプレイ再生できる

仕組み(全体像)

ユーザー作りたいユニットを日本語で説明
→
仕様相談 AI質問と提案で仕様を確定(SpecAgent)
→
実装 AI行動スクリプト+モデル構成を生成(CodeAgent)
→
自動検査静的検証・テスト・アセット照合
→
シミュレーション決定論対戦 → リプレイ+バランス評価
  • ここからこの流れを3つの工程に分けて、実物で見ていきます
  • 全工程がモック実行でも再現可能(API なしで動くデモも用意)。途中の成果物はすべて構造化データとして保存

工程 1相談 — 質問と候補カードで仕様を固める

  • あいまいな依頼には質問を返す。勝手に進まない(質問ゲート)
  • 迷う場面では「おすすめ/攻めた案/変化球」の3案を、見返り・リスク付きのカードで提示 — 人は選ぶだけ
  • 確定した仕様は右側の仕様シート(構造化データ)に固まっていく。この対話は軽量モデルで成立

ユニットのかたち — 「頭脳」と「技の中身」の2層

ユニットの動きは2層でできています。生成するのは下の層だけです:

ユニット AI(頭脳) いつ動く・誰を狙う・どの行動を出すかを毎瞬決める。
7種の既製テンプレートから選ぶだけ — 生成しない(例: Kiter=距離を取って撃つ)
行動の指示書(生成物) AI が「今スキル1」と決めた瞬間に呼ばれ、その技が具体的に何をするかを返す。
LLM が生成するのはこの指示書10枚 — 出撃時・移動・通常/強攻撃・スキル×2・被弾・転倒・死亡

戦い方の頭脳は検証済みの既製品。生成の自由度は「技の中身」に限定されるので、棒立ちや迷走が構造的に起きない

指示書のやることは1つだけ:

  • 状況メモ(ctx)を見る — 自分の位置とHP割合、AI が決めた攻撃対象、スキルが使えるか
  • 命令カード(16種)から数枚選んで返す — 「範囲減速」「クールダウン開始」など
  • 実際に世界を動かすのはシミュレーター側。指示書は命令を返すだけで、直接は何もできない

次のスライドのコードは、この読み方だけで追えます:
「skill1: … = スキルを使うときの指示書」「return [...] = 返す命令カードの束」

工程 2生成 — 1回でファイル一式が出てくる

export const unitScript: UnitScriptV1 = {
  version: 1,
  init: () => [
    commands.equip('chain_snare_tool'),
    commands.equip('rider_lance_polearm'),
    commands.mount('void_horse')
  ],
  …(move / turn / basicAttack 略)…
  skill1: (ctx: ActionContext) => {
    if (!ctx.cooldowns.ready('skill1')) return [];
    const output: Command[] = [
      ctx.commands.areaSlow({
        center: ctx.intent.targetPosition ?? ctx.self.position,
        radius: 4,
        slowRatio: 0.5,
        duration: 2,
        vfx: 'chain_tether_spark'
      }),
      ctx.commands.startCooldown('skill1', 7)
    ];
    return output;
  },

スライド3の依頼文に対して、軽量モデル(flash)が初回一発で出したコードの抜粋(無修正)

  • script.ts — 行動スクリプト(左)。equip / mount の ID はすべてアセット台帳の実在 ID
  • manifest.json — アクション定義(射程・クールダウン・対象選択・効果範囲の予算)
  • composition.json — ボクセルの見た目構成(体・槍・騎乗)
  • tests.generated.ts — 形状テスト(構造の自己検証)
  • brief.ja.md — 日本語の設計解説

注目してほしいのは賢さではなく語彙の狭さ。決められたコマンドと実在 ID しか書けない形に誘導している

読み方(前スライド): init=出撃時の指示書、skill1=スキルの指示書、ready('skill1')=「もう使える?」の確認、return [...]=返す命令カードの束

実演(約10分)

いま見た工程1〜3を、ライブで通します。注目ポイント:

  • ① 人の仕事は「選ぶ」と「見極める」 — AI が質問し、候補カードで提案。人は方向を選ぶだけ
  • ② 生成物はすべて証拠として残る — スクリプト・アセット・検査結果が構造化データで保存される
  • ③ 完成の定義が「検証を通ること」 — できたユニットはその場で対戦し、結果が数値になる

運用の要点①「原型箱」— 型を先に作る

LLM に自由にコードを書かせない。型に流し込ませる。この実証で用意した型と、現場での対応物:

コマンド API(16種)⇔社内のスキル・アクション定義。生成コードはこのメニューを返すことしかできない
データスキーマ(23本)⇔マスタデータ定義。検証を通らないデータは次の工程に進めない
素材カタログ(43点)⇔アセット管理台帳。実在する ID しか参照できない
参照ドキュメント(6本)⇔通った実例1本+レビューで弾いた理由の記録(失敗パターン集)

新規開発というより、すでにある資産を「LLM が読める契約」に固め直す作業が中心だった

運用の要点①部品箱の中身 — 実際の型定義

できること = Command 型(抜粋。全16種):

export type Command =
  | { type: 'meleeArc'; damage: number;
      radius: number; angle: number;
      damageType: DamageType; … }
  | { type: 'areaSlow'; center: Vec2;
      radius: number; slowRatio: number;
      duration: number; vfx?: string }
  | { type: 'applyStatus';
      targetUnitId: string;
      status: StatusId; duration: number; … }
  | …equip / mount / moveToward /
    areaDamage / playAttributeVfx など

見えるもの = ActionContext 型(全部):

export interface ActionContext {
  self: UnitSelfView;      // 自分の位置とHP比率
  intent: ActionIntent;    // 攻撃対象・目標地点
  commands: CommandFactory; // 左の16種
  scalars: { attack: number;
             moveSpeed: number };
  cooldowns: { ready(id: string): boolean };
  rng: { next(): number };  // シード付き乱数
}

敵の一覧も、HP の生値も、座標系の詳細も渡さない。語彙と視界を絞るほど、軽量モデルの合格率が上がった

運用の要点②自動の受け入れ検査 — 通らないものは入らない

  • サンドボックス: 生成コードはゲーム状態に触れない。コマンド列を返すだけ
  • 静的検査: 禁止 import・存在しないアセット参照・記述漏れを、実行前に機械が却下(工程3で見た実録)
  • 決定論シミュレーション: 同じ条件なら同じ結果。生成物の良し悪しを再現可能な実験として判定できる
  • 失敗の資産化: 弾いた理由は失敗パターン集に蓄積 → 次の生成プロンプトに自動反映

人手レビューを増やすのではなく、検査を仕組みにする。これが安いモデルを使う前提条件

運用の要点③モデルは役割で使い分ける

仕様の対話→軽量級(flash)で成立。全構成で共通
コード生成(現在の型)→軽量単独でも 10/10。未見のお題でも同じ結果。中量級(pro)でも 10/10
同じ軽量モデル(初期の型)→25% — 差を作ったのはモデルではなく、型と失敗パターン集の成熟
  • コストの実測: 受け入れ1体あたり約1.4万トークン(入力1.15万/出力0.25万)。flash と pro でトークン量はほぼ同じなので、コスト差は単価差そのもの — 軽量化の削減がまるごと効く
  • 検査が壊れた生成物を全部止めるので、「どこまで安くできるか」の実験がノーリスク
  • 初期に重量級へ切り替えても数字は動かなかった — モデルの賢さは、型の不備を埋めない

情報の扱い(社内導入で先に整理する論点)

  • 外部 API に送るのはプロンプト(仕様の相談内容+型の定義)だけ。ソースコード全体やマスタデータ一式は送らない
  • 接続先は OpenAI 互換 API で、設定1つで差し替え可能 — 1.5B の小型オープンモデル+llama.cpp で、仕様相談〜生成〜検証の全工程を実機確認済み
  • 生成・検査・評価の証拠ファイルはすべて手元に残る — 「AI が何をしたか」を後から全件監査できる
  • API なしのモック実行モードがあり、ネットワーク遮断環境でもデモ・CI が回る

どこから始めるか

「作らせて検証する」が向くのは、出力がデータやスクリプトの形に落ちるコンテンツ:

  • スキル・行動スクリプト、装備・敵などのマスタデータ、配置・エンカウント、バランス初期値
  • 条件は3つ: 機械検証できる/失敗が安くて速い/人の仕事を仕様の対話とレビューに寄せられる
  • 投資順序は「型 → 検査 → 軽量モデル」。モデル選定は最後

検証の土台(この実証での決定論シミュレーション)がない領域は、生成より先に土台づくりから
